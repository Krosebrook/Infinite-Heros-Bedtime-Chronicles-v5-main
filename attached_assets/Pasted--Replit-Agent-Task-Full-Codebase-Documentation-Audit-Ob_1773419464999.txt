# Replit Agent Task: Full Codebase & Documentation Audit

## Objective

Perform a comprehensive audit of this project. Analyze the actual codebase, compare it against any existing documentation, identify drift between docs and reality, then generate senior-grade documentation that reflects the true current state. Do not guess. Read every file.

## Phase 1 — Codebase Audit

Scan the entire repository. For each of the following, document what actually exists (not what should exist):

### 1.1 Project Structure
- Map the full directory tree (exclude `node_modules`, `.git`, build artifacts)
- Identify the framework, language, and runtime (versions from `package.json`, `requirements.txt`, `.python-version`, `replit.nix`, or equivalent)
- List all entry points (server start, build commands, scripts)
- Identify environment configuration files (`.env.example`, `.replit`, `replit.nix`, `vercel.json`, `wrangler.toml`, etc.)

### 1.2 Architecture
- Map all routes/endpoints (REST, GraphQL, WebSocket — whatever exists)
- Identify the data layer (database, ORM, direct queries, migrations)
- Map authentication and authorization flows (auth provider, session handling, middleware)
- Document all external service integrations (APIs, SDKs, webhooks)
- Identify background jobs, cron tasks, or queue workers if present
- Map state management approach (client-side and server-side)

### 1.3 Dependencies
- List all production dependencies with versions
- List all dev dependencies with versions
- Flag any dependencies with known vulnerabilities (`npm audit` or equivalent)
- Flag any outdated major versions (compare against latest stable)
- Identify unused dependencies (imported but never referenced in code)

### 1.4 Security Posture
- Check for hardcoded secrets, API keys, tokens, or credentials in any file (including comments and dead code)
- Verify environment variables are used for all sensitive values
- Check for input validation on external boundaries (API routes, form handlers, URL params)
- Check for SQL injection vectors (raw queries without parameterization)
- Check for XSS vectors (unsanitized output rendering)
- Check authentication middleware coverage (which routes are protected, which are open)
- Check CORS configuration
- Check rate limiting presence
- Check security headers (HSTS, CSP, X-Frame-Options, X-Content-Type-Options)
- If using Supabase: verify Row-Level Security (RLS) is enabled on all tables containing user data

### 1.5 Testing
- Identify test framework and configuration
- Count total tests, passing, failing, skipped
- Calculate approximate code coverage (or note if coverage tooling is missing)
- Identify untested critical paths (auth, payments, data mutations)
- Run the test suite and capture results

### 1.6 Code Quality
- Identify linting configuration (ESLint, Prettier, Ruff, etc.) — present or missing
- Check TypeScript strict mode (if TypeScript project)
- Flag any `any` types, `@ts-ignore`, or `eslint-disable` overrides
- Identify dead code (exported but never imported, unreachable branches)
- Check error handling patterns (try/catch coverage, error boundaries, graceful degradation)
- Check logging patterns (structured vs console.log, log levels)

## Phase 2 — Documentation Audit

Find and read ALL existing documentation in the repo:

### 2.1 Inventory
- `README.md` — exists? complete? accurate?
- `CONTRIBUTING.md` — exists?
- `CHANGELOG.md` — exists? up to date?
- `API.md` or equivalent API docs — exists?
- Inline code comments — density and quality
- JSDoc / docstrings — coverage
- `.env.example` — exists? matches actual env vars used in code?
- Architecture diagrams or decision records — present?
- Any `/docs` directory or wiki content

### 2.2 Drift Analysis
For every piece of documentation found, compare it against the actual codebase:

- **Endpoints documented but not implemented** — list them
- **Endpoints implemented but not documented** — list them
- **Environment variables documented but not used** — list them
- **Environment variables used but not documented** — list them
- **Dependencies documented but removed** — list them
- **Dependencies added but not documented** — list them
- **Features described in README that don't exist in code** — list them
- **Features in code not mentioned anywhere in docs** — list them
- **Stale instructions** (references to old commands, deprecated config, wrong file paths) — list them

## Phase 3 — Generate Documentation

Based on what you actually found (NOT what should exist in theory), generate the following documents. Every claim must be verifiable against the codebase. Zero placeholders. Zero aspirational content.

### 3.1 README.md (Complete Rewrite)
Structure:
```
# Project Name
One-line description of what this actually does.

## Tech Stack
Exact frameworks, languages, versions — from package.json/lockfiles, not assumptions.

## Prerequisites
What must be installed to run this locally. Exact versions.

## Setup
Step-by-step from clone to running. Every command. Tested.

## Environment Variables
| Variable | Required | Description | Where to Get It |
Every env var actually used in the code. No extras. No missing ones.

## Available Scripts
Every script in package.json (or equivalent) with description of what it does.

## Project Structure
Directory tree with 1-line descriptions of what each major folder/file does.

## API Reference
Every endpoint: method, path, auth required, request body, response shape.
(If > 20 endpoints, create separate API.md and link to it.)

## Architecture Overview
How the pieces connect. Data flow. Auth flow. External integrations.
Use text descriptions — do not generate image files.

## Testing
How to run tests. What's covered. Known gaps.

## Deployment
How this gets deployed. What environment. Any CI/CD configuration.

## Known Issues
Current bugs, tech debt, or limitations you found during audit.
```

### 3.2 .env.example
Generate from actual environment variable usage in the codebase. Include:
- Every `process.env.X` or equivalent reference found in code
- Descriptive comments for each variable
- Safe placeholder values (not real secrets)

### 3.3 ARCHITECTURE.md
- System diagram (described in text/markdown — component names, data flow directions, integration points)
- Database schema summary (tables, key relationships, RLS status if Supabase)
- Auth flow (step-by-step: login → token → session → refresh → logout)
- External service map (which services, what data flows to/from each)
- Error handling strategy (how errors propagate, where they're caught, how they're reported)

### 3.4 SECURITY.md
- Current security posture (what's in place)
- Vulnerabilities found during audit (with severity: Critical / High / Medium / Low)
- Remediation recommendations (specific, not generic — reference exact files and line numbers)
- Dependency vulnerability report

### 3.5 AUDIT-REPORT.md
- Summary of all findings from Phase 1 and Phase 2
- Categorized by: Critical (fix now) / High (fix this sprint) / Medium (fix this quarter) / Low (backlog)
- Each finding includes: what's wrong, where it is (file + line), why it matters, how to fix it
- Documentation drift table (what docs say vs what code does)

## Execution Rules

1. **Read the codebase first.** Do not generate anything until you've scanned every file.
2. **Do not rewrite existing code.** This is an audit — observe and document, don't refactor.
3. **Do not create files outside the `/docs` directory** (except README.md and .env.example in root).
4. **Flag uncertainty.** If you can't determine something from the code alone, say "UNVERIFIED: [reason]" — do not guess.
5. **Be specific.** Reference exact file paths, line numbers, function names. No vague "some files have issues."
6. **Run the test suite** if one exists. Include actual output.
7. **Run dependency audit** (`npm audit`, `pip audit`, or equivalent). Include actual output.
8. **Compare against best practices** but prioritize documenting reality over prescribing ideals.
